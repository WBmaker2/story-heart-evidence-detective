# 이야기 마음 근거 수사대 MVP 구현 계획

**목표:** 초등 2학년 학생이 독창적인 짧은 이야기를 읽고, 작품 속 인물의 행동·표정·대사를 근거로 가능한 마음이나 생각을 추론하며, 서로 다른 해석도 각각 작품 속 근거가 있으면 타당할 수 있음을 배우는 서버 없는 교육용 웹앱을 구현합니다.

**아키텍처:** 별도 저장소 `story-heart-evidence-detective`에 Vite, React, TypeScript 기반 정적 앱을 구성합니다. 안내 이야기 1편과 고정 사건 6편, 검수된 마음 어휘, 근거 카드, 복수 해석 지도를 정적 데이터로 제공합니다. 학생은 마음 카드 1개와 근거 카드 2개를 연결하며, 순수 함수가 사람이 검수한 근거 조합을 기준으로 `근거가 잘 이어짐`, `일부만 이어짐`, `다른 해석으로도 이어짐`, `근거가 부족함`을 결정적으로 판정합니다. 단일 정답, 생성형 AI, 자유 감정 일기, 서버 저장은 사용하지 않습니다.

**기술 스택:** Vite, React, TypeScript, CSS, SVG, Vitest, React Testing Library, Playwright, `@axe-core/playwright`, Node.js

**현재 상태:** 이 문서는 구현 계획만 정의합니다. 별도 구현 요청 전에는 새 저장소 생성, 패키지 설치, 원고·삽화 제작, 코드 작성, 테스트 실행, 커밋, 배포, HVC 등록을 하지 않습니다.

### 성취수준 적용 원칙

- A·B·C 성취수준을 학생 점수, 난이도 이름, 등급으로 노출하지 않습니다.
- `마음이나 생각을 짐작한다`를 마음 낱말 맞히기로 축소하지 않고 작품 안의 관찰 가능한 근거와 연결합니다.
- 근거 선택은 성취기준 원문의 별도 채점 요소가 아니라, 짐작의 이유를 보이게 만드는 수업용 발판으로 사용합니다.
- 앱은 작품 속 인물의 마음을 확정하거나 실제 사람의 마음을 읽는 능력을 평가하지 않습니다.
- 자기 경험과의 비교, 자유 말하기, 창작 표현은 교사 수업과 후속 활동 영역으로 남깁니다.

## 2. 제품 접근 비교와 결정

| 접근 | 방식 | 장점 | 위험 | 결정 |
|---|---|---|---|---|
| 검수된 복수 해석 지도 | 마음별로 사람이 확인한 근거 2개 조합을 정적 데이터에 선언하고 학생이 마음 1개·근거 2개를 연결 | 복수 답을 동등하게 인정하고 오프라인에서도 결정적으로 검증 가능 | 원고와 가능한 조합을 사전에 정성껏 검수해야 함 | **MVP 채택** |
| 단일 정답 감정 퀴즈 | 이야기마다 감정 정답 1개를 고름 | 구현과 채점이 단순함 | 문학 해석을 감정 이름 맞히기로 축소하고 사용자 요구와 충돌 | 사용하지 않음 |
| 자유 서술·키워드 채점 | 학생이 마음과 이유를 직접 쓰고 낱말 포함 여부로 판정 | 표현 자유도가 높음 | 맞춤법·유의어·저학년 입력 부담 때문에 타당한 답을 놓침 | MVP 제외 |
| 생성형 AI 채점 | 자유 문장을 모델이 평가 | 예상하지 못한 해석을 받을 수 있음 | 결과 변동, 개인정보·비용·네트워크, 저학년 설명 책임 문제 | 후속 연구만 가능 |

### 추천안의 핵심 이유

- `correctMindId` 하나가 아니라 `reviewedReadings` 2~3개를 두어 해석의 복수성을 데이터 구조부터 보장합니다.
- 학생은 자유 글쓰기 대신 마음 카드와 본문 근거를 골라 2학년 수준에서 추론 관계에 집중합니다.
- 근거 2개 연결은 교육과정의 정답 조건이 아니라 여러 단서를 함께 살펴보는 앱 활동 규칙으로만 설명합니다.
- 같은 종류 근거 2개도 제출할 수 있으며 사람이 검수한 정확한 조합인지에 따라 판정합니다. `표정+표정`도 서로 다른 관찰을 담고 이야기 맥락과 함께 두 검토자가 타당성을 확인한 경우에만 등록합니다.
- 검수된 해석과 일치할 때까지 추측을 강요하지 않고, 완성된 추론을 제출해 피드백을 읽으면 다음 이야기로 갈 수 있습니다.
- 서버, 로그인, 카메라, 마이크, 생성형 AI 없이 한 기기에서 안정적으로 진행할 수 있습니다.

## 3. 기존 앱과 계획의 차별화

카탈로그 56개와 기존 계획에는 작품 속 한 인물의 행동·표정·대사를 연결하여 복수의 타당한 마음을 추론하는 앱이 없습니다.

| 기존 앱·계획 | 겹칠 수 있는 요소 | 이야기 마음 근거 수사대의 차별화 기준 |
|---|---|---|
| 말놀이 퐁당! 감정 라디오 | 감정 어휘와 국어 활동 | 자기 감정을 고르고 낭송·녹음하는 활동이 아니라 작품 속 인물의 마음을 본문 근거로 추론합니다. |
| 우리 반 마음 날씨 예보관 | 마음·감정 카드 | 학생 자신의 아침 감정을 기록하거나 학급 감정 현황을 모으지 않습니다. |
| 상상 톡톡, 이어지는 이야기 기차 | 이야기와 인물 | 협업 글쓰기나 AI 장면 생성을 하지 않고 완결된 원작을 읽고 해석합니다. |
| 뚝딱뚝딱 문장 만들기 공장 | 저학년 문장 카드 | 단어 순서와 문장 부호가 아니라 인물의 마음과 작품 속 근거의 연결을 다룹니다. |
| 팩트체크 편집국 계획 | 근거를 고르는 활동 | 실제 정보의 사실성을 판정하지 않고 문학 작품에서 여러 해석이 근거와 함께 공존할 수 있음을 학습합니다. |
| AI 데이터 품질검사소 계획 | 복수 기준 검토 | 데이터 오류를 찾지 않고 이야기의 인물·시점·마음·근거 관계를 읽습니다. |

### 차별화 유지 규칙

- 핵심 순환은 `이야기 읽기 → 인물과 바로 이때 확인 → 마음 1개 선택 → 근거 2개 선택 → 근거 연결 피드백 → 다른 해석 살펴보기`입니다.
- `오늘 내 기분`, `내가 이런 감정을 느낀 경험`, 이름·학번·학급, 자유 감정 일기를 묻지 않습니다.
- 이야기 생성, 이어 쓰기, 녹음, 공유 게시판, AI 그림 생성 기능을 넣지 않습니다.
- 감정을 착함·나쁨으로 평가하거나 인물의 실제 마음을 확정하지 않습니다.
- 한 근거만 보고 `웃음=기쁨`, `시선 회피=거짓말`, `말이 없음=슬픔`처럼 고정 대응하지 않습니다.

## 4. MVP 제품 계약

### 기본 정보

| 항목 | 결정 |
|---|---|
| 앱 이름 | 이야기 마음 근거 수사대 |
| 주 대상 | 초등 2학년, 1~2학년군 국어 읽기·문학 활동 |
| 1회 활동 시간 | 15~18분 |
| 콘텐츠 | 안내 이야기 1편과 고정 순서 원작 사건 6편 |
| 이야기 길이 | 사건당 4~6문장, 35~70어절, 문장당 12어절 이하 우선 |
| 추론 범위 | 한 사건에서 한 초점 인물과 한 시점 |
| 마음 카드 | 사건당 4개, 느낌과 생각·바람을 말로 구분 |
| 근거 카드 | 사건당 5~6개, 행동·표정·대사 중 자연스러운 2종류 이상 |
| 학생 선택 | 마음 1개와 서로 다른 카드 2개, 근거 종류는 같아도 제출 가능 |
| 복수 해석 | 일반 사건마다 검수된 타당한 마음 2~3개 |
| 판정 | 정오·점수 없이 근거 연결 정도 4종과 미완성 안내 |
| 진행 조건 | 완성된 추론을 제출하고 피드백을 확인하면 판정 종류와 무관하게 다음 사건 가능 |
| 결과 | 점수 없는 읽기 전용 `수사 기록판` |
| 저장 | React 메모리만 사용하며 새로고침 시 시작 화면으로 초기화 |
| 네트워크 | 초기 정적 자원 외 API·외부 요청 없음 |
| 개인정보 | 이름·학번·학급·감정 일기·사진·음성·자유 서술 입력 없음 |

### 학생 학습 목표

학생은 활동 후 다음을 할 수 있어야 합니다.

1. 이야기에서 누구의 어느 때 마음을 짐작하는지 확인합니다.
2. 마음을 직접 볼 수 없으며 행동·표정·대사가 마음을 짐작하는 단서가 될 수 있음을 이해합니다.
3. 마음 카드 하나와 이야기 속 근거 카드 두 개를 연결합니다.
4. 자신의 선택을 `~한 마음일 수 있어요. 왜냐하면 ~했기 때문이에요.`라는 문장으로 확인합니다.
5. 같은 인물과 시점에도 근거에 따라 서로 다른 마음 해석이 가능함을 확인합니다.
6. 표정 하나나 이야기 밖의 고정관념만으로 마음을 단정하지 않습니다.
7. 선택한 근거만으로 마음과의 연결을 충분히 설명하기 어려울 수 있음을 이해합니다.

### 학생 활동 흐름

1. **시작:** `마음은 눈에 보이지 않아요. 이야기 속 단서로 생각해 볼 수 있어요.`를 확인합니다.
2. **안내 사건:** 마음 하나와 근거 두 개를 버튼으로 연결하는 방법을 연습합니다.
3. **이야기 읽기:** 전체 보기 또는 한 문장씩 보기로 4~6문장을 읽습니다.
4. **수사 대상 확인:** `누구의 마음인가요?`, `바로 언제인가요?`를 한 문장으로 확인합니다.
5. **마음 선택:** 4개 카드 중 이야기 근거로 설명해 보고 싶은 마음 하나를 고릅니다.
6. **근거 선택:** 행동·표정·대사 카드 중 서로 다른 카드 두 개를 고릅니다. 종류가 같아도 제출할 수 있습니다.
7. **수사 보고서:** 선택으로 자동 완성된 근거 문장을 읽고 `근거 연결 살펴보기`를 누릅니다.
8. **연결 피드백:** 현재 마음과 두 근거가 어떻게 이어지는지 확인합니다.
9. **다른 해석:** 검수된 다른 마음과 근거 연결 하나를 `다른 생각도 가능해요`에서 살펴봅니다.
10. **다음 사건:** 수정할 수 있고, 피드백을 확인했다면 맞히기 반복 없이 다음으로 이동합니다.
11. **수사 기록판:** 해 본 이야기와 사용한 근거 종류만 정리하고 점수·정답 수는 표시하지 않습니다.

### 학생용 용어 계약

- 화면에서는 `마음`, `생각·바람`, `단서`, `근거`, `가능한 해석`, `바로 이때`를 사용합니다.
- `정답`, `오답`, `틀린 감정`, `마음 읽기`, `거짓말 탐지`, `감정 분석`은 사용하지 않습니다.
- 느낌 카드와 생각·바람 카드는 라벨로 구분하되 우열을 두지 않습니다.
- 피드백은 `실제로 이런 마음이었어요`가 아니라 `이 근거로 이렇게 생각할 수 있어요`라고 씁니다.
- `근거는 언제나 두 종류여야 해요`라고 가르치지 않고 `오늘은 두 단서를 함께 살펴보는 연습을 해요`라고 안내합니다.
- 앱에 준비되지 않은 해석도 문학적으로 가능할 수 있으므로 판정 범위를 `현재 이야기와 제공된 근거 카드 안에서`로 제한합니다.

### MVP에서 제외

- 자기 감정 기록, 자기 경험 회상 질문, 일기, 상담·심리검사 기능
- 자유 서술, 맞춤법 검사, 키워드 채점, 음성 입력, 발음 평가
- 생성형 AI 이야기·그림·피드백 생성과 외부 모델 호출
- 카메라 표정 인식, 생체 정보, 마이크, 녹음, 사진 업로드
- 드래그 앤 드롭, 스와이프, 길게 누르기, 캔버스 포인터 판정
- 문제 자동 생성, 무작위 순서, 시간제한, 목숨, 점수, 별, 배지, 순위, 연속 성공
- 교사 문제 제작기, 학생별 진도 저장, 학급 코드, 교사 대시보드
- 로그인, 브라우저 저장소, 분석 로그, 서버 데이터베이스
- 배포, HVC 등록, 공유 게시판, 인쇄·다운로드

## 5. 복수 해석과 피드백 계약

### 복수 타당 답 원칙

- 일반 사건에는 검수된 `reviewedReadings`를 2~3개 둡니다.
- 각 해석은 마음 하나와 사람이 함께 읽어 타당성을 확인한 `reviewedEvidencePairs` 2개 이상을 가집니다.
- 각 검수 조합은 정확히 근거 카드 2개와 두 근거가 함께 마음을 뒷받침하는 설명을 가집니다. 같은 마음에 개별적으로 연결된 근거라도 등록되지 않은 교차 조합은 자동으로 타당하다고 보지 않습니다.
- 별도의 `correctMindId`, `singleAnswer`, `acceptedMindIds`를 두지 않고 타당한 마음 목록은 `reviewedReadings`에서 파생합니다.
- 하나의 근거 조합이나 근거 카드가 여러 해석에 함께 쓰일 수 있으며, 각 해석에는 다른 해석과 구별되는 조합도 하나 이상 둡니다.
- 학생은 마음 하나만 고르지만 피드백에서 고르지 않은 다른 타당한 해석을 확인합니다.
- 검수 지도에 없는 해석을 `절대 불가능`이라고 말하지 않고 현재 제공된 근거와의 연결이 약하다고만 설명합니다.

### 판정 우선순위

| 코드 | 판정 조건 | 학생 피드백 방향 | 다음 이동 |
|---|---|---|---|
| 제출 전 안내 | 마음 미선택, 근거 2개 미만, 같은 카드 중복 | 빠진 선택을 한 가지만 안내하고 판정 함수를 호출하지 않음 | 선택 보완 후 제출 |
| `supported` | 두 근거의 정규화된 조합이 선택 마음의 검수 조합과 정확히 일치함 | `두 근거를 함께 보면 이 마음이라고 생각할 수 있어요.` | 다른 해석 확인 후 수정 또는 다음 |
| `evidence-link-mismatch` | 정확한 근거 조합이 다른 검수 마음에 등록되어 있음 | `이 두 근거로는 {다른 마음}이라고도 생각할 수 있어요. 지금 고른 마음과 이어지는 다른 근거도 살펴볼까요?` | 수정 또는 예시 확인 후 다음 |
| `partially-supported` | 선택 마음의 검수 조합과 한 카드만 겹치고, 다른 마음의 정확한 조합과 일치하지 않음 | `한 근거는 잘 이어져요. 두 근거를 함께 보며 연결을 다시 살펴보세요.` | 수정 또는 예시 확인 후 다음 |
| `insufficient-evidence` | 현재 선택으로 어느 검수 해석도 충분히 뒷받침되지 않음 | `현재 근거카드만으로는 이 마음과의 연결을 충분히 확인하기 어려워요.` | 수정 또는 예시 확인 후 다음 |

- 완성된 추론은 네 판정 코드 모두에서 피드백과 검수 해석 예시를 확인할 수 있습니다.
- 같은 종류의 서로 다른 카드 2개도 완성된 추론입니다. 종류만으로 제출을 막거나 오답으로 처리하지 않습니다.
- `supported`를 받을 때까지 선택지를 바꾸게 하지 않습니다.
- `partially-supported`, `evidence-link-mismatch`, `insufficient-evidence`도 학습 자료이며 실패 횟수로 저장하지 않습니다.
- 알 수 없는 마음·근거 ID나 잘못된 콘텐츠는 학생 답으로 처리하지 않고 이름 있는 `학습 자료 오류`로 중단합니다.

### 자동 완성 수사 보고서

- 자유 입력 없이 `나는 {인물}이 {마음 문장형} 수 있다고 생각해요. {근거 1}과 {근거 2}가 단서예요.`를 만듭니다.
- 마음 어휘는 조사와 서술어가 자연스럽도록 `label`과 `sentenceForm`을 분리합니다.
- 근거 문구는 본문에서 추출하며 같은 문장을 데이터에 다시 타이핑하지 않습니다.
- 학생이 선택하지 않은 해석은 제출 전 공개하지 않습니다.

### 수사 기록판

- `행동 단서를 찾아봄`, `표정 단서를 살펴봄`, `대사 단서를 연결함`, `다른 해석을 확인함`을 읽기 전용으로 표시합니다.
- 학생이 선택한 마음과 근거는 현재 세션 안에서만 사건별로 다시 볼 수 있습니다.
- 판정 코드, 정답 개수, 성공률, 시도 횟수, 속도, 순위를 표시하지 않습니다.
- `다시 수사하기`는 확인 뒤 같은 고정 순서와 빈 상태로 시작합니다.

## 6. 원작 이야기 은행 계획

### 원고 작성 계약

- 안내 이야기 1편과 일반 사건 6편을 이 앱을 위해 새로 작성합니다.
- 사건당 4~6문장, 35~70어절, 문장당 12어절 이하를 우선하며 등장인물은 최대 2명입니다.
- 추론 대상 인물과 시점은 `바로 이때` 한 문장으로 분명히 제시합니다.
- 마음 후보는 4개, 근거카드는 5~6개이며 이야기에서 자연스럽게 드러나는 근거 종류를 2종류 이상 둡니다.
- 표정은 `슬픈 표정`처럼 해석하지 않고 `입술을 꼭 다물었다`, `눈이 조금 커졌다`처럼 관찰 사실로 씁니다.
- 타당한 마음 낱말과 직접적인 유의어를 본문에 쓰지 않아 답을 직접 노출하지 않습니다.
- 반어, 빈정거림, 복잡한 거짓말, 관용구, 이중부정, 성인 관계 갈등을 사용하지 않습니다.
- 교과서·동화·방송·영화의 인물, 사건, 문장, 유명 장면을 각색하지 않습니다.
- 실제 학생·학교 사건, 실명, 학급 자료를 원고 소재로 사용하지 않습니다.

### 원작 설계 골격

아래 표는 구현 단계에서 완성할 원고의 골격이며 최종 전문이 아닙니다.

| 순서 | 내부 ID·가제 | 원작 상황 골격 | 추론할 바로 이때 | 검수할 복수 해석 | 핵심 근거 조합 |
|---:|---|---|---|---|---|
| 안내 | `tutorial-wet-invitation`·젖은 초대장 | 정성껏 만든 초대장이 빗물에 젖자 인물이 말리고 글씨를 다시 살핌 | 초대장을 물에서 건진 직후 | 속상한 마음, 초대받을 친구를 걱정하는 마음 | 표정+행동, 행동+대사 |
| 1 | `second-start-line`·두 번째 출발선 | 이어달리기 연습 중 바통을 떨어뜨린 인물이 친구들을 보고 다시 달림 | 바통을 주워 다시 출발하는 순간 | 친구들이 걱정되는 마음, 다시 해 보고 싶은 마음, 끝까지 이어 가고 싶은 마음 | 행동+대사, 행동+행동 |
| 2 | `window-paper-star`·창가의 종이별 | 친구 작품이 가운데 자리에 뽑히자 박수치고 자신의 작품도 옆에 붙여도 되는지 물음 | 친구 작품을 축하한 뒤 자기 작품을 바라볼 때 | 친구가 잘해 기쁜 마음, 자기 작품에 대한 아쉬운 마음, 함께 전시하고 싶은 마음 | 행동+대사, 표정+행동 |
| 3 | `stopped-robot`·로봇이 멈춘 순간 | 발표 중 로봇이 멈추자 인물이 멈춘 위치를 묻고 점검표를 펼침 | 점검표를 펼쳐 함께 살피기 시작하는 순간 | 이유가 궁금한 마음, 해결하고 싶은 마음, 발표를 이어 가고 싶은 마음 | 대사+행동, 행동+행동 |
| 4 | `waiting-flowerpot`·기다리는 화분 | 친구 화분에는 잎이 나고 자기 화분은 아직 흙뿐이어서 번갈아 보고 다음 관찰 날짜를 표시함 | 두 화분을 살핀 뒤 기록할 때 | 언제 싹이 날지 궁금한 마음, 조금 아쉬운 마음, 기대하며 기다리는 마음 | 표정+행동, 행동+대사 |

| 6 | `quiet-applause`·조용한 박수 | 발표자가 고른 축하 방식을 확인한 인물이 손을 흔들고 들은 점을 말함 | 발표가 끝나 선택된 방식으로 축하할 때 | 발표가 인상 깊어 기쁜 마음, 응원하고 싶은 마음, 발표자의 선택을 존중하고 싶은 마음 | 상황을 담은 대사+행동, 표정+대사 |

### 사건별 데이터 수용 기준

- 마음 후보 4개 중 검수된 해석은 2~3개이며 나머지는 황당한 오답이 아니라 현재 근거가 약한 선택입니다.
- 각 검수 해석에는 사람이 확인한 근거 2개 조합이 2개 이상이며, 조합 전체에 고유 근거 카드 3개 이상이 포함됩니다.
- 서로 다른 해석이 공유하는 근거 카드 또는 조합이 1개 이상, 각 해석만의 독점 조합이 1개 이상 있어야 합니다.
- 어떤 검수 조합에도 들어가지 않는 중립 근거가 1개 이상 있어야 합니다.
- 근거 종류가 같다는 이유만으로 조합을 금지하지 않습니다. 다만 `표정+표정`은 두 표현이 같은 의미를 반복하지 않고 이야기 맥락까지 함께 읽을 때 성립한다고 두 검토자가 확인한 경우에만 등록합니다.
- 모든 근거는 해당 이야기 본문의 정확한 글자 범위를 가리킵니다.
- 6개 사건 전체에서 검수 해석 카드의 위치를 분산하여 위치가 정답 단서가 되지 않게 합니다.
- 행동·표정·대사의 균형은 사건마다 기계적으로 맞추지 않고 이야기 은행 전체에서 각 종류가 일반 사건 4편 이상에 자연스럽게 등장하는지 확인합니다.
- 삽화는 새로운 결정적 정보를 추가하지 않고 본문에 있는 관찰 사실만 시각화합니다.

### 원고·해석 독립 검수

1. 작성자는 1차 검토자에게 원고·초점 인물·시점만 제공하고 마음 후보와 해석 지도를 가립니다.
2. 초등 저학년 지도 경험이 있는 검토자 2명이 각자 가능한 마음과 정확한 근거 조합을 자유롭게 찾습니다.
3. 2차 검토에서 마음 후보·검수 조합·피드백을 공개하고, 마음 후보 4개와 선택 가능한 모든 근거 2개 조합의 분류표를 보며 누락, 편견, 어휘 적합성, 억지 조합을 확인합니다.
4. 의도한 해석을 두 검토자가 작품 문구로 설명할 수 있는지 확인합니다.
5. 한 검토자라도 등록되지 않은 타당한 해석이나 조합을 제시하면 지도에 추가하거나 원고를 고치며, 타당하지만 미등록인 조합이 없다고 두 검토자가 확인해야 검수를 끝냅니다.
6. 근거가 약한 후보를 두 검토자가 모두 타당하다고 보면 학생 답을 탓하지 않고 원고·후보·검수 조합을 수정합니다.
7. 1·2차 결과, 원고 버전, 검토자 역할, 검토일을 함께 기록하고 원고 변경 뒤 텍스트 앵커, 사건·은행 `contentVersion`, 검수 기록, 업데이트 내역을 갱신합니다.
8. 출시 전 2학년 소규모 사용성 확인은 정답률이 아니라 인물·시점·근거 선택 가능 여부만 집계형 관찰 메모로 확인합니다.
9. 사용성 확인은 학교·보호자 절차를 따르고 실명·학번을 적지 않으며 사진·음성·화면을 녹화하지 않습니다. 집계 메모의 보관 기간과 폐기일을 미리 정합니다.

### 마음 어휘 계약

- 전체 기본 마음 어휘는 느낌과 생각·바람을 합쳐 약 12~14개로 제한합니다.
- 카드 하나에는 느낌 또는 생각·바람 하나만 담고 `신기하고 궁금함`처럼 두 개념을 한 카드에 묶지 않습니다.
- 각 낱말에는 2학년이 읽을 수 있는 한 문장 뜻풀이를 제공합니다.
- `아쉽다/속상하다`처럼 가까운 말은 같은 사건에 함께 제시하지 않거나 둘 다 타당한 해석으로 검수합니다.
- 가까운 낱말의 미세한 차이를 약한 연결의 유일한 이유로 사용하지 않습니다.
- 이모지나 표정 그림을 마음 낱말의 주된 정의로 사용하지 않습니다.

## 7. 공정성·저작권·개인정보 원칙

### 마음 추론의 공정성

- 같은 종류의 근거 2개도 동등하게 검토합니다. `표정+표정`은 단일 표정을 감정과 일대일 대응하거나 같은 의미를 반복한 조합은 제외하고, 독립적인 두 관찰과 이야기 맥락을 두 검토자가 확인한 조합만 등록합니다.
- 성별, 피부색, 옷, 체형, 사투리, 한국어 숙련도, 보조기기, 가족 형태를 마음 근거로 사용하지 않습니다.
- 눈 맞춤, 큰 목소리, 빠른 말하기, 조용함을 정직성·예의·지능과 연결하지 않습니다.
- 웃음·눈물·침묵 같은 표현 하나를 고정 감정과 일대일 대응하지 않습니다.
- 장애가 있는 인물을 다른 인물의 친절을 보여 주는 장치나 문제 상황으로 사용하지 않습니다.
- `착한 마음`, `나쁜 마음`으로 감정을 채점하지 않고 작품 속 근거 연결만 검토합니다.
- 따돌림, 폭력, 죽음, 질병, 가족 갈등, 공개 망신처럼 정서 부담이 큰 소재는 MVP에서 제외합니다.

### 저작권·출처

- 각 이야기에는 `origin: 'original-for-this-app'`, 작성일, 콘텐츠 버전, 검수 체크리스트 ID를 기록합니다.
- 원고 데이터의 `externalSourceUrls`는 빈 배열이어야 하며 유명 작품을 변형한 초안은 폐기합니다.
- 독창성은 자동 검사만으로 증명할 수 없으므로 사람의 유사성·출처 검토를 완료 조건으로 둡니다.
- 삽화·아이콘·글꼴·효과음은 직접 제작하거나 재배포 가능한 라이선스만 사용하고 `THIRD_PARTY_NOTICES.md`에 기록합니다.
- AI 보조 초안을 사용할 경우 살아 있는 작가의 문체 모사를 금지하고 사람이 문장·유사성·학년 적합성을 다시 검수합니다.

### 개인정보·안전

- 학생 이름, 반, 번호, 실제 경험, 현재 감정, 사진, 음성, 자유 텍스트를 수집하지 않습니다.
- `너라면 언제 이런 마음이 들었나요?` 같은 자기 경험 입력 질문을 MVP에 넣지 않습니다.
- 카메라 표정 분석, 감정 진단, 상담 추천, 위험도 판정 기능을 만들지 않습니다.
- `localStorage`, 쿠키, 분석 도구, 외부 전송 없이 현재 탭 메모리에서만 동작합니다.
- 앱이 학습 자료이며 실제 사람의 마음이나 심리 상태를 판단하는 도구가 아니라는 안내를 시작 화면에 둡니다.

## 8. 드래그 없는 상호작용과 접근성

### 조작 계약

- 마음은 `<fieldset>`·`<legend>`와 같은 `name`의 네이티브 라디오, 근거는 네이티브 체크박스로 선택합니다.
- 모든 핵심 기능을 Tab·Shift+Tab·방향키·Space·Enter·터치로 완료합니다.
- 근거 체크박스는 최대 2개이며 세 번째 선택은 기존 선택을 유지하고 이유를 인접 문장으로 안내합니다.
- 같은 종류의 서로 다른 카드 2개도 제출할 수 있습니다. 근거 종류의 다양성은 추천 힌트와 콘텐츠 구성으로 경험하게 하며 제출 조건으로 강제하지 않습니다.
- 드래그, 스와이프, 길게 누르기, 포인터 이동량, 캔버스 히트 테스트를 사용하지 않습니다.
- 모든 클릭·터치 대상은 최소 48×48 CSS px로 만듭니다.

### 글 읽기와 시각 등가 표현

- 본문은 기본 20px 이상, 줄간격 1.7, 한 줄 25~35자 안팎으로 제한합니다.
- `전체 이야기`와 `한 문장씩 보기`를 제공하되 어느 모드에서도 같은 텍스트와 같은 근거를 사용합니다.
- 추론 단계에서도 본문과 `바로 이때` 문장을 화면에 계속 남깁니다.
- 원작 SVG 장면은 보조 자료이며 모든 표정·행동 관찰 사실을 본문 텍스트에도 제공합니다.
- 대체 텍스트는 `슬퍼 보이는 인물`이 아니라 `입술을 다물고 젖은 종이를 바라보는 인물`처럼 관찰 사실만 설명합니다.
- 근거카드는 `대사 근거, 3번 문장: ...`처럼 유형·문장 번호·실제 문구를 함께 읽게 합니다.
- 선택 상태는 색뿐 아니라 체크 표시, 굵은 테두리, `선택됨` 문구로 구분합니다.

### 초점·알림·모달

- 새 사건은 이야기 제목, 단계 이동은 새 단계 제목에 프로그래밍 방식으로 초점을 옮깁니다.
- 라디오 선택은 네이티브 방향키 동작을 보존하고 체크박스는 Space로 전환합니다.
- 제출 뒤 결과 제목으로 초점을 옮기되 제목 자체를 live 영역에 넣지 않습니다. 별도 `role="status" aria-live="polite"`는 짧은 상태만 한 번 알려 중복 낭독을 막습니다.
- 자동으로 다음 사건으로 넘기지 않고 학생이 `피드백 살펴봤어요`를 누르게 합니다.
- 업데이트 내역·초기화 모달은 이름, 초점 가두기, Escape 닫기, 호출 버튼 초점 복귀를 지원합니다.
- 오류 문구는 관련 fieldset과 `aria-describedby`로 연결합니다.

### 반응형·모션·검수

- 320px·375px에서는 이야기, 마음, 근거, 보고서를 한 열로 배치하고 가로 스크롤을 만들지 않습니다.
- 200% 확대와 1280px 화면의 400% 확대(동등한 320 CSS px 리플로)에서도 본문, 선택 상태, 피드백, 초점 테두리가 잘리거나 겹치지 않습니다.
- 사용자가 줄간격·문자간격·단어간격을 늘려도 본문과 버튼 문구가 잘리거나 겹치지 않습니다.
- 일반 텍스트 대비는 4.5:1 이상, 큰 텍스트·UI 경계·초점 표시는 3:1 이상을 유지합니다.
- `prefers-reduced-motion: reduce`에서는 카드 전환 애니메이션을 제거합니다.
- forced-colors에서도 라디오·체크박스·근거 유형·초점이 구분되어야 합니다.
- 자동 재생 소리, 필수 효과음, 번쩍임, 화면 흔들기, 시간제한을 사용하지 않습니다.
- `@axe-core/playwright`를 보조 게이트로 사용하고 macOS VoiceOver와 실제 수업용 태블릿의 iOS VoiceOver 또는 Android TalkBack 중 하나로 읽기→라디오→체크박스→피드백→모달 흐름을 검수합니다.

## 9. 도메인·데이터·상태 계약

### 핵심 데이터 모델

아래 형식은 구현 단계의 기준 계약입니다. 이야기의 타당한 해석은 단일 정답 ID가 아니라 `reviewedReadings`의 관계로 표현합니다.

```ts
type EvidenceKind = 'action' | 'expression' | 'dialogue';
type MindKind = 'feeling' | 'thought-or-wish';

type ReviewCode =
  | 'supported'
  | 'evidence-link-mismatch'
  | 'partially-supported'
  | 'insufficient-evidence';

type CompletionIssueCode =
  | 'mind-required'
  | 'two-distinct-evidence-cards-required';

interface MindDefinition {
  id: string;
  label: string;
  sentenceForm: string;
  kind: MindKind;
  studentFriendlyMeaning: string;
}

interface StoryParagraph {
  id: string;
  text: string;
}

interface TextAnchor {
  paragraphId: string;
  startOffset: number;
  endOffset: number;
  exactQuote: string;
}

interface EvidenceCard {
  id: string;
  kind: EvidenceKind;
  anchor: TextAnchor;
}

interface ExplanationDefinition {
  id: string;
  kind:
    | 'evidence-rationale'
    | 'evidence-pair-summary'
    | 'interpretation-summary';
  text: string;
}

interface ReviewedEvidencePair {
  id: string;
  evidenceCardIds: readonly [string, string];
  evidenceRationaleIds: readonly [string, string];
  pairSummaryId: string;
}

interface ReviewedReading {
  id: string;
  mindId: string;
  reviewedEvidencePairs: readonly [
    ReviewedEvidencePair,
    ReviewedEvidencePair,
    ...ReviewedEvidencePair[],
  ];
  interpretationSummaryId: string;
}

interface ReviewedContentRegistry {
  minds: readonly MindDefinition[];
  explanations: readonly ExplanationDefinition[];
  reviewChecklistIds: readonly string[];

}

  origin: 'original-for-this-app';
  createdAt: string;

}

  schemaVersion: 1;
  contentVersion: string;
  id: string;
  order: number;
  title: string;

    StoryParagraph?,

  readingId: string;
```

### 데이터 설계 원칙

- `exactQuote`는 UI 표시용 복사본이 아니라 앵커 손상을 찾는 검수용 값입니다. `paragraph.text.slice(startOffset, endOffset) === exactQuote`가 반드시 성립해야 합니다.
- 근거카드 화면 문구는 매번 본문의 텍스트 앵커에서 파생하여 원고와 근거가 서로 달라지지 않게 합니다.
- 마음별 허용 목록은 별도 필드로 복사하지 않고 `reviewedReadings.map(reading => reading.mindId)`에서 파생합니다.
- 타당성은 개별 근거 링크의 합집합이 아니라 `reviewedEvidencePairs`의 정확한 2개 조합으로 선언합니다.
- 조합 속 근거 ID는 사건의 `evidenceCards` 선언 순서로 정규화하고, 학생의 선택 순서는 판정이나 설명 순서에 영향을 주지 않습니다.
- `evidenceRationaleIds`, `pairSummaryId`, `interpretationSummaryId`는 중앙 설명 레지스트리를 가리키며 콘텐츠 안에 긴 중복 설명을 넣지 않습니다.
- 레지스트리 종류도 함께 검사합니다. `evidenceRationaleIds`는 `evidence-rationale`, `pairSummaryId`는 `evidence-pair-summary`, `interpretationSummaryId`는 `interpretation-summary`만 참조합니다.
- `InferenceReview.rationaleIds`는 정규화된 근거 1 설명→근거 2 설명→조합 설명 순서를 유지합니다.
- 마음 후보 4개에는 검수 해석 2~3개와 현재 근거가 약한 후보가 함께 들어갑니다.
- 사건 삽화·검수 체크리스트 ID는 레지스트리에 실제로 존재해야 하고, 삽화는 정적 SVG만 가리키며 외부 URL을 허용하지 않습니다.
- 후보에 있지만 `reviewedReadings`에 없는 마음은 타당 조합이 없는 후보로 처리합니다. 후보에도 없는 마음이나 사건에 없는 근거 ID는 빈 결과로 삼지 않고 이름 있는 `ContentDomainError`로 중단합니다.
- 데이터에 `correctMindId`, `answer`, `isCorrect`, `score`, `points`, `attempts`, `streak` 필드를 만들지 않습니다.

### 순수 함수 경계

| 함수 | 책임 | 주요 불변 조건 |
|---|---|---|
| `deriveEvidenceQuote(caseData, evidenceId)` | 본문 앵커에서 근거 문구와 문장 번호를 파생 | 저장된 별도 문구를 사용하지 않음 |
| `deriveReviewedMindIds(caseData)` | 검수된 마음 ID를 해석 목록에서 파생 | 중복 제거, 원래 순서 보존 |
| `getReviewedEvidencePairsForMind(caseData, mindId)` | 선택 마음의 검수 조합 반환 | 후보·미검수 마음은 빈 배열, 후보 밖 ID는 자료 오류 |
| `canonicalizeEvidencePair(ids, caseData)` | 두 근거 ID를 사건 선언 순서로 정렬 | 학생 선택 순서와 무관한 안정적 비교 |
| `completeInference(draft, caseData)` | 마음 1개·서로 다른 카드 2개와 참조 ID를 검증 | 미완성은 `CompletionResult`, 알 수 없는 ID는 자료 오류, 근거 종류는 제한하지 않음 |
| `reviewInference(inference, caseData, registry)` | 정확한 검수 조합으로 `InferenceReview` 반환 | 완성 입력만 받고 동일 입력은 항상 동일 결과 |
| `buildInvestigationSentence(inference, caseData, registry)` | 선택으로 학생용 보고서 문장 생성 | 자유 텍스트·HTML 삽입 없음 |
| `getAlternativeReading(inference, review, caseData)` | 판정과 일치하는 다른 검수 해석을 선택 | 불일치 판정 우선, 그 외 선언 순서, 무작위 없음 |
| `validateTextAnchors(caseData)` | 모든 범위·문구·문단 참조 검사 | 범위를 벗어나거나 달라지면 빌드 실패 |
| `validateStoryCase(caseData, registry)` | 사건별 구조·교육·모든 참조 계약 검사 | 오류마다 사건 ID와 규칙 이름 제공 |
| `validateStoryBank(bank)` | 안내 1편·일반 6편과 레지스트리·전체 분포 검사 | 순서·ID·버전·후보 위치 확인 |
| `deriveSessionSummary(records, bank)` | 사건·근거 ID를 원자료와 대조해 점수 없는 기록판 항목 생성 | 판정 코드·시도 횟수 미노출, 알 수 없는 ID는 자료 오류 |

### 판정 알고리즘

제출 경계와 판정 경계를 분리합니다.

1. `completeInference`가 마음 선택, 서로 다른 카드 2개, 후보 마음 ID, 사건 근거 ID를 확인합니다. 미선택·중복은 제출 전 안내로 돌려보내고, 알 수 없는 ID는 `ContentDomainError`로 중단하며 어느 경우에도 `reviewInference`를 호출하지 않습니다.
2. 같은 종류 근거 2개도 완성된 추론으로 인정하고, 선택한 두 ID를 사건의 근거 선언 순서로 정규화합니다.
3. `reviewInference`가 정규화한 조합과 모든 `reviewedEvidencePairs`를 선언 순서대로 비교합니다.
4. 선택 마음의 정확한 조합과 일치하면 `supported`입니다. 같은 조합이 다른 마음에도 등록되어 있으면 전체 일치 목록에는 보존하되 선택 마음을 대표 해석으로 둡니다.
5. 선택 마음에는 없고 다른 마음의 정확한 조합과 일치하면 `evidence-link-mismatch`입니다. 여러 마음에 동시에 일치하면 `reviewedReadings` 선언 순서의 첫 항목을 대표로 삼고 전체 ID를 같은 순서로 보존합니다.
6. 정확한 다른 조합 일치가 없고 선택 마음의 검수 조합 하나와 카드 1개만 겹치면 `partially-supported`입니다. 여러 조합과 겹치면 선언 순서의 첫 조합과 사건 카드 순서의 첫 근거를 설명에 사용합니다.
7. 그 밖에는 `insufficient-evidence`입니다.

`getAlternativeReading`은 `evidence-link-mismatch`에서 실제 대표 일치 해석을 먼저 보여 줍니다. 나머지 판정에서는 학생이 고르지 않은 첫 검수 해석을 선언 순서대로 선택합니다. 이 규칙으로 피드백이 가리키는 해석과 아래의 `다른 생각도 가능해요` 예시가 어긋나지 않게 합니다.

### 이야기 은행 자동 검증

`npm run check:content`는 최소한 다음 규칙을 검사하고 하나라도 어기면 종료 코드 1로 실패해야 합니다.

- 안내 사건 1편과 일반 사건 6편, 고정된 `order`, 전체 ID 유일성
- 사건당 4~6문장, 35~70어절, 권장 문장 길이 초과 목록
- 서로 다른 마음 후보 정확히 4개, 근거 5~6개, 서로 다른 마음의 검수 해석 2~3개
- 사건당 자연스러운 근거 종류 2종류 이상, 은행 전체에서 행동·표정·대사가 각각 일반 사건 4편 이상에 등장
- 해석당 검수 조합 2개 이상과 조합 전체의 고유 근거 카드 3개 이상
- 조합마다 서로 다른 카드 정확히 2개와 정규화 순서, 같은 해석 안의 정규화 중복 조합 금지
- 조합 ID는 은행 전체에서 고유해야 하며, 서로 다른 해석이 같은 정규화 근거 조합을 공유하는 것은 허용
- `표정+표정` 조합에는 서로 다른 관찰·비중복 의미·이야기 맥락을 확인한 두 검토자의 검수 기록 필수
- 해석끼리 공유하는 카드 또는 조합 1개 이상, 해석별 독점 조합 1개 이상, 어느 조합에도 없는 중립 근거 1개 이상
- 후보·해석·근거·문단·조합·설명 ID 유일성, 후보에 없는 검수 마음 금지
- 모든 마음·설명·검수 체크리스트·정적 삽화 참조 ID의 레지스트리 존재
- 사건 전체에서 검수 마음 카드 위치가 한 자리로 치우치지 않는지 확인
- 본문에 검수 마음 낱말이 직접 들어가지 않는지 금칙어 스캔 후, 유의어는 사람 검수 목록으로 별도 확인
- 전체 마음 어휘 12~14개, 카드당 단일 개념, 가까운 낱말의 억지 구분은 사람 검수
- 일반 사건 최소 1편에 서로 다른 마음이 공유하는 정확한 조합, 최소 1편에 검수된 `행동+행동` 또는 `대사+대사` 조합 존재
- 위 공유 조합·같은 종류 조합의 사건·마음·조합 ID를 안정적인 E2E 픽스처 상수로 공개
- `origin`, 작성일, 빈 `externalSourceUrls`, 검수 체크리스트 ID, 사건·은행 콘텐츠 버전 존재
- 단일 정답·점수·시도 횟수·무작위·시간·네트워크 의존 필드 금지

### 앱 상태와 상태 전이

학생이 아직 제출하지 않은 선택과 제출한 기록을 섞지 않도록 사건 상태는 판별 가능한 유니온으로 만듭니다.

```ts
type CaseAttempt =
  | {
      phase: 'reading';
      readingMode: 'all' | 'sentence-by-sentence';
      paragraphIndex: number;
    }
  | {
      phase: 'mind';
      draft: { mindId: null; evidenceCardIds: readonly [] };
    }
  | {
      phase: 'evidence';
      draft: {
        mindId: string;
        evidenceCardIds: readonly string[];
      };
      noticeCode: CompletionIssueCode | 'max-two-evidence' | null;
    }
  | {
      phase: 'review';
      submitted: CompletedInference;
      review: InferenceReview;
      alternativeReadingId: string;
    };

interface CaseRecord {
  caseId: string;
  inference: CompletedInference;
}

type AppState =
  | { screen: 'start'; overlay: OverlayState }
  | {
      screen: 'investigation';
      caseCursor: 'tutorial' | 0 | 1 | 2 | 3 | 4 | 5;
      attempt: CaseAttempt;
      records: readonly CaseRecord[];
      overlay: OverlayState;
    }
  | {
      screen: 'summary';
      records: readonly CaseRecord[];
      overlay: OverlayState;
    };

type OverlayState = null | 'update-history' | 'reset-confirm';
```

| 이벤트 | 허용 상태 | 전이와 보존 규칙 |
|---|---|---|
| `START_TUTORIAL` | 시작 | 안내 사건의 `reading`으로 이동 |
| `SET_READING_MODE` | 읽기 | 같은 본문과 앵커를 유지하고 표시 방식만 변경 |
| `NEXT_SENTENCE`·`PREVIOUS_SENTENCE` | 한 문장씩 읽기 | 범위 안에서 문장 위치만 변경 |
| `FINISH_READING` | 읽기 | 초점 인물·시점 확인 후 `mind`로 이동 |
| `SELECT_MIND` | 마음·근거 | 선택 마음을 가진 `evidence` 상태로 이동하거나 변경, 기존 근거는 유지하고 안내는 지움 |
| `TOGGLE_EVIDENCE` | 근거 | 정상 토글은 안내를 지움, 세 번째는 선택 목록을 보존하고 `max-two-evidence`만 설정 |
| `SUBMIT_INFERENCE` | 근거 | 카드 종류와 무관하게 완성 여부·참조를 검증하고 성공하면 `review`로 이동, 미완성이면 선택을 보존하고 안내 코드 설정 |
| `REVISE_INFERENCE` | 검토 | 제출한 마음·근거를 그대로 가진 `evidence`로 돌아감 |
| `ACKNOWLEDGE_REVIEW` | 검토 | 안내 사건이면 기록 없이 1번 사건으로 이동하고, 일반 사건이면 기록을 한 번 추가한 뒤 판정 코드와 무관하게 다음 사건 또는 요약으로 이동 |
| `OPEN_OVERLAY`·`CLOSE_OVERLAY` | 모든 화면 | 학습 상태를 변경하지 않고 모달만 열고 닫음 |
| `REQUEST_RESET`·`CONFIRM_RESET` | 모든 화면 | 확인 후 모든 메모리 상태를 비우고 시작으로 이동 |

- MVP는 완료한 사건 재방문을 지원하지 않으므로 일반 사건 기록을 정확히 한 번만 추가합니다.
- `CaseRecord`에는 사건 ID와 최종 추론만 저장하고 근거 종류와 다른 해석 확인 표시는 이야기 원자료와 완료 흐름에서 파생합니다.
- 사건 이동은 `supported` 여부를 조건으로 삼지 않습니다.
- `REVISE_INFERENCE`는 선택을 지우지 않아 비교하며 고칠 수 있게 합니다.
- 새로고침 복구, 세션 직렬화, 날짜·시간, 난수, `localStorage`를 리듀서에 넣지 않습니다.

## 11. 구현 작업 순서

아래 작업은 **추후 구현 요청을 받은 뒤에만** 별도 저장소에서 수행합니다. 각 작업은 실패하는 테스트 또는 검증 규칙을 먼저 추가하고, 최소 구현으로 통과시킨 뒤 정리합니다. 커밋명은 제안일 뿐이며 현재 문서 작성 단계에서는 실행하지 않습니다.

### 작업 1. 정적 앱 골격과 품질 게이트

**예정 파일:** `package.json`, `vite.config.ts`, `tsconfig.json`, `eslint.config.js`, `playwright.config.ts`, `src/main.tsx`, `src/App.tsx`, `src/test/setup.ts`, `scripts/check-file-lengths.mjs`, `README.md`

1. Vite React TypeScript 최소 골격과 오류 경계를 구성합니다.
2. `dev`, `build`, `preview`, `lint`, `typecheck`, `test`, `test:e2e`, `check:content`, `check:file-lengths`, `verify` 스크립트를 명시합니다.
3. 파일 길이 검사 자체의 통과·실패 테스트를 먼저 만듭니다.
4. 앱 셸에는 제목과 비어 있는 콘텐츠 자리만 두며 사건 UI는 아직 만들지 않습니다.
5. `npm run typecheck && npm test && npm run build && npm run check:file-lengths` 통과 후 다음 작업으로 이동합니다.

**미래 커밋 예시:** `chore: scaffold story evidence detective`

### 작업 2. 복수 해석 도메인과 순수 판정 함수

**예정 파일:** `src/domain/types.ts`, `evidence-anchors.ts`, `inference-review.ts`, `investigation-sentence.ts`, `tests/domain/*.test.ts`

1. 타입과 테스트 픽스처를 정의하되 단일 정답 필드는 만들지 않습니다.
2. 앵커 추출의 정상·범위 초과·문구 불일치 테스트를 작성합니다.
3. 제출 전 `CompletionResult`와 네 가지 완성 `InferenceReview`를 분리하여 테스트합니다.
4. 사람이 등록한 정확한 근거 조합만 `supported`이며, 같은 마음에 속한 근거라도 등록되지 않은 교차 조합은 자동 인정되지 않음을 확인합니다.
5. 후보지만 미검수인 마음과 후보 밖 마음, 사건 밖 근거 ID의 처리가 각각 빈 조합과 `ContentDomainError`로 구분되는지 테스트합니다.

## 13. 역할 분담과 오케스트레이션 원칙

- `gpt-5.6-sol`은 이 계획의 범위·교육 설계·작업 순서·완료 판단 같은 핵심 기획과 오케스트레이션에만 사용합니다.
- 구현 승인을 받은 뒤 실제 코드·원고 데이터 연결·테스트·리뷰는 `gpt-5.6-terra`가 담당합니다.
- 구현 에이전트가 병렬 하위 작업을 나눌 때는 `gpt-5.6-luna`를 사용하고, 원고 검수와 코드 검수 결과를 구현 담당이 다시 통합 확인합니다.
- 원고 작성과 독립 해석 검수는 코드 작업과 분리하며, 검수되지 않은 원고를 임시 정답 데이터로 굳히지 않습니다.
- 각 작업은 앞 작업의 테스트와 콘텐츠 게이트가 통과한 뒤 시작합니다. 특히 작업 3의 원고·해석 검수 완료 전에는 작업 5~7의 사건별 UI를 마감하지 않습니다.
- 이 계획 문서만 현재 `vibecoding-lab` 저장소에 둡니다. 실제 앱은 별도 `story-heart-evidence-detective` 저장소에 만들며, 작업 시작 전에 경로와 쓰기 권한을 다시 확인합니다.
- 구현이 시작된 뒤 범위나 해석 모델이 바뀌면 먼저 이 계획 또는 대상 저장소의 설계 문서를 갱신한 뒤 코드를 수정합니다.

- [ ] 이야기마다 한 인물과 한 시점, 서로 다른 마음 4개, 자연스러운 2종류 이상의 근거 5~6개가 있습니다.
- [ ] 같은 종류 근거도 제출할 수 있고, `표정+표정`은 독립적인 두 관찰과 이야기 맥락을 두 검토자가 확인한 조합만 등록합니다.
- [ ] 각 마음 후보와 선택 가능한 모든 근거 2개 조합을 검토표에서 분류하여 타당하지만 미등록인 조합이 없음을 두 검토자가 확인합니다.

### 복수 해석·학습 경험

- [ ] 단일 정답 필드와 정오·점수·시도·순위 UI가 없습니다.
- [ ] 두 가지 이상의 검수된 해석과 사람이 등록한 정확한 근거 조합을 실제 E2E에서 각각 타당하게 인정합니다.
- [ ] 같은 마음의 개별 근거를 임의로 교차한 미검수 조합은 자동으로 타당 처리되지 않습니다.
- [ ] `supported`가 아니어도 피드백을 확인하면 다음 사건으로 이동할 수 있습니다.
- [ ] 다른 해석은 제출 뒤에만 보이며 작품 속 근거와 함께 제시됩니다.
- [ ] 앱이 다루지 못한 해석을 불가능하다고 단정하지 않습니다.
- [ ] 최종 화면은 학습 행동만 요약하고 학생 능력이나 감정을 등급화하지 않습니다.

### 개인정보·안전·접근성

- [ ] 자기 감정·실제 경험·이름·학급·자유 글·사진·음성을 입력하거나 저장하지 않습니다.
- [ ] 서버·외부 API·분석 도구·쿠키·브라우저 저장소 없이 정적 자원만 사용합니다.
- [ ] 드래그 없이 키보드·터치·스크린리더로 전체 활동을 완료합니다.
- [ ] 48px 조작 대상, 320px 반응형, 200%·400% 확대, 사용자 글자 간격, reduced motion, forced colors 검수를 통과합니다.
- [ ] SVG 없이도 본문만으로 모든 근거를 판단할 수 있습니다.
- [ ] axe 중대 위반 0건과 데스크톱·수업용 태블릿 스크린리더 수동 흐름 점검을 완료합니다.

### 기술·운영

- [ ] 모든 코드·스타일·스크립트·SVG 파일이 500줄 미만이며 자동 검사로 강제합니다.
- [ ] 타입·린트·단위·상태·컴포넌트·콘텐츠·E2E·빌드 검사가 모두 통과합니다.
- [ ] 고정 이야기·고정 순서·결정적 판정이 새로고침마다 동일합니다.
- [ ] 업데이트 내역 버튼과 실제 개발일·개선일·간단한 변경 내용이 있습니다.
- [ ] 콘솔 오류, 깨진 앵커, 유효하지 않은 참조 ID, 외부 요청이 없습니다.
- [ ] 실제 구현·배포·HVC 등록은 각각 사용자의 별도 요청과 검증 뒤에만 완료로 처리합니다.

## 15. MVP 이후 확장 후보

MVP 검증 전에는 아래 기능을 넣지 않습니다.

- **근거가 정말 부족한 사건:** 어떤 마음도 확정하기 어려운 원작을 별도 유형으로 설계하여 `잘 모르겠다`를 타당한 결론으로 다룹니다.
- **마음 변화 시간선:** 같은 인물의 시작·중간·끝 마음을 서로 다른 근거와 연결합니다.
- **교사용 토의 모드:** 학생 데이터를 수집하지 않고 한 화면에 여러 검수 해석과 토의 질문만 표시합니다.
- **읽어 주기:** 저작권과 접근성을 확인한 사전 녹음 음원 또는 브라우저 보조 기능을 선택 제공하되 자동 재생·음성 저장은 하지 않습니다.
- **3~4학년 확장:** 별도 교육과정 검토 뒤 인물·사건·배경의 관계와 작품 감상을 다루는 더 긴 이야기로 확장합니다.
- **교사 원고 묶음:** 로컬 파일 불러오기만 허용하고 스키마·앵커·복수 해석·편견 검수 게이트를 모두 통과한 묶음만 실행합니다.

자기 감정 기록, 실제 사람의 표정 분석, 감정 진단, 자유 서술 AI 채점은 확장 후보에도 포함하지 않습니다.
