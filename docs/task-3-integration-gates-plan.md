# Task 3 통합 게이트 및 릴리스 정리 계획

## 범위

- 기존 제품 동작이나 콘텐츠 범위를 바꾸지 않고 정적 검사와 릴리스 문서만 보강한다.
- 소스 파일 길이, 개인정보·저장소·외부 런타임 요청, 학생용 점수·정답 표현, 콘텐츠 앵커, 서버 렌더링 HTML을 자동 검증한다.
- 키보드 포커스, 320px 반응형, 모션 감소, 강제 색상, 터치 영역 요구가 코드와 테스트에 연결되어 있는지 고정한다.

## 구현 순서

1. 새 검사 모듈의 기대 동작을 테스트로 먼저 작성하고 실패(RED)를 기록한다.
2. 파일 길이 검사와 정적 릴리스 감사를 작은 스크립트로 구현한다.
3. `typecheck`, `lint`, `test`, `check:content`, `check:file-lengths`, `verify` 스크립트를 하나의 실용적인 검증 흐름으로 연결한다.
4. README와 편집 상태 문서에 실행법, 개인정보 보호 모델, 결정적 해석 모델, 교육과정 부분 정렬의 한계, 미완료 수동 검증을 명시한다.
5. `npm run verify`와 새 `npm run build`를 실행하고 전체 변경을 자체 리뷰한 뒤 커밋한다.

## 완료 조건

- 모든 코드·스타일·스크립트·SVG 파일이 500줄 미만이다.
- 스타터 메타데이터·스켈레톤 UI·`codex-preview`가 제품에 남지 않는다.
- 제품 코드에 외부 런타임 요청과 `localStorage`, `sessionStorage`, 쿠키 사용이 없다.
- 자동 게이트가 모두 통과하며, 사람 2인의 콘텐츠 검수와 실제 기기 스크린리더·태블릿 검증은 완료로 주장하지 않는다.

## 위험과 비범위

- 문자열 기반 정적 검사는 런타임 브라우저·보조공학 검증을 대신하지 않는다.
- 배포, HVC 등록, 인증, 분석, 저장, AI, 외부 서비스는 추가하지 않는다.

## 리뷰 후속 수정 계획

1. 정적 앱에 쓰이지 않는 D1 예제·DB·Drizzle 설정과 의존성을 제거하되 `.openai/hosting.json`의 `d1`·`r2` null 계약과 Sites 플러그인은 보존한다.
2. `vite.config.ts`에서 D1 스타터 이름과 자리표시자를 제거하고, 정적 앱에 필요한 최소 워커 바인딩만 남긴다.
3. 릴리스 감사 테스트에 Next 서버 쿠키 API, Cookie 헤더 접근, 워커 외부 URL·요청, D1·Drizzle 스타터 잔여물 회귀 사례를 RED로 추가한다.
4. 워커의 동일 출처 `env.ASSETS.fetch(new Request(new URL(path, request.url)))`만 명시적으로 허용하고 다른 워커 네트워크 요청은 실패시킨다.
5. 집중 테스트의 GREEN 뒤 전체 `verify`와 별도 새 `build`를 실행하고, 수동 사람·기기 검증은 계속 미완료로 기록한다.
